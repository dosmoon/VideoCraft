# LLM 本地部署技术选型文档

**版本**:2026-05(v2)  
**适用范围**:本地部署、开源、可商用、配合 ASR 输出做下游处理  
**配套文档**:云端 API 选型(另文)

---

## 1. 选型目标

承接 ASR 选型的硬约束(本地部署 / 开源 / 可商用),并附加 4 个具体任务:

| 任务 | 输入 | 输出 | 关键能力需求 |
|---|---|---|---|
| 翻译 | 字幕条目 / 段落 | 目标语言文本 | 多语言(中英西法德俄)、口语化 |
| 字幕处理 | ASR 原始转录 | 修正后的字幕 | 错别字纠正、断句、标点;速度优先 |
| 热点段落分析 | 完整 transcript | 高能片段及其时间戳 | 长上下文、内容理解、主观判断 |
| 长视频分段 | 完整 transcript | 章节结构 JSON | 长上下文、结构化输出 |

语言优先级:**英语 > 中文 > 西班牙语 / 法语 / 德语 / 俄语**。

---

## 2. 用户硬件分布(本次选型的关键前提)

自媒体创作者的真实硬件分布(粗略估计,Steam Hardware Survey + 常识):

| 硬件档位 | 占比估计 | 典型配置 |
|---|---|---|
| 8 GB VRAM 及以下 | ~40% | RTX 3050 / 3060 / 4060、集成显卡、Apple M1-M3 base 8GB |
| 12-16 GB VRAM | ~30% | RTX 3060 12GB / 4060 Ti / 4070 / Apple M-series 16GB |
| 16-24 GB VRAM | ~15% | RTX 4070 Ti / 3090 / Apple M-series 24-36GB |
| 24 GB+ VRAM | ~5% | RTX 4090 / 5090 / Apple M-series 48GB+ |
| 仅 CPU / 无独显 | ~10% | 老笔记本 / 入门 Mac |

**核心结论**:**70% 以上的用户跑不动 16GB 以上的模型**。

这意味着选型必须以**轻量本地模型**为默认,而不是追求"最强本地模型"。复杂任务(高质量翻译、热点分析)的本地方案对大多数人都不现实——这部分应该交给云端 API,见配套文档。

| 模型族 | 英语 | 中文 | 西/法/德 | 俄语 |
|---|---|---|---|---|
| Qwen3.x 系列 | 强 | **最强(母语)** | 强 | 中等偏上 |

| Mistral Small 4 | 强 | 中等 | 强(法语母语) | 中等 |

中文场景 Qwen 几乎无悬念,欧语场景 Gemma 4 略胜,俄语任何模型都不强。

## 5. 关键技术事实:为什么 4B 是合理的默认

很多人对"4B 模型"有过低的预期,认为它是"勉强能用"的级别。事实上 2025-2026 年这一代 4B 模型(Qwen3-4B-Instruct-2507、Qwen3.5-4B、Gemma 4 E4B)在以下任务上表现已经很好:

- **字幕错别字修正、标点恢复、断句优化**:几乎无差距,4B 和 30B 输出相同
- **常见语言对的字幕级翻译**(中英、英西、英法、英德):约 80-85% 场景质量够用
- **JSON 结构化输出**:配合 Grammar 约束,4B 完全胜任
- **简单总结、关键词提取**:够用

4B 模型在以下任务上明显不足:

- **长尾语言对**(俄语、阿拉伯语、小语种)
- **专业术语翻译**(医学、法律、金融)
- **跨文化语境保留**(俚语、网络用语、典故)
- **完整 transcript 的热点段落判断**(主观能力依赖大模型)
- **超长上下文**(>32K)的稳定召回

这个能力边界**完美对应于"什么应该走本地,什么应该走云端"的分界线**——简单、高频、批量的任务走本地,质量敏感的任务推用户走云端。

## 6. 推荐架构(三档)

### 默认档:Qwen3-4B-Instruct-2507(覆盖 70%+ 用户)

| 项 | 数值 |
|---|---|
| 显存(Q4_K_M) | ~3 GB |
| 总下载体积 | ~2.5 GB |
| 协议 | Apache 2.0 |
| 上下文 | 128K(YaRN 扩展可到 256K) |
| RTX 3060 速度 | ~80-100 tok/s |
| Apple M2 base 速度 | ~30-40 tok/s |
| 1 小时字幕翻译耗时 | 1-3 分钟(主流硬件) |

**适用任务**:
- 字幕处理(全场景,质量与大模型几乎无差)
- 中英简单互译
- 西/法/德 ↔ 英 的常规翻译
- 短 transcript 总结(<10 分钟视频)

**不适用**:
- 俄语翻译(质量明显下降,推荐云端)
- 长视频章节分段(质量取决于上下文长度)
- 热点段落主观判断(需要更大模型)

### 可选高端档:Qwen3.6-35B-A3B(给 24GB+ 用户)

| 项 | 数值 |
|---|---|
| 显存(Q4_K_M) | ~21 GB |
| 总下载体积 | ~20 GB |
| 协议 | Apache 2.0 |
| 上下文 | 256K 原生 |
| RTX 4090 速度 | 100-120 tok/s |
| 1 小时字幕翻译耗时 | 3-5 分钟 |

**核心特性**:MoE 架构,激活仅 3B,**速度比 9B Dense 还快,质量是 35B 级别**。

**为什么不选 9B Dense**:9B 在 4090 上速度约 100 tok/s,与 35B-A3B 持平甚至略慢;而质量差距约 30%。9B 这个尺寸在 MoE 普及后已经没有清晰的最优适用区间。

**适用任务**:全部 4 个任务,质量全面优于 4B。

**目标用户**:有 RTX 3090 / 4090 / 5090 的少数玩家、Apple M-series 32GB+ 统一内存用户。

### 极轻量档:Gemma 4 E2B 或 Qwen3-1.7B(给 8GB 以下硬件)

针对老笔记本、Apple M1 8GB、纯 CPU 跑的用户。

| 选项 | 显存 | 用途 |
|---|---|---|
| Qwen3-1.7B | ~1 GB | 字幕处理(其他任务建议走云端) |
| Gemma 4 E2B | ~1.5 GB | 字幕处理 + 简单翻译,带原生工具调用 |

**重要提醒**:这个档位不要尝试做翻译和分析,效果会让用户对本地模型留下负面印象。给清晰提示"高质量任务建议启用云端"。

## 7. 三档配置总览

| 档位 | 模型 | 显存 | 默认下载 | 复杂任务建议 |
|---|---|---|---|---|
| **极轻量** | Qwen3-1.7B 或 Gemma 4 E2B | <2 GB | 否(可选) | 走云端 |
| **默认** | Qwen3-4B-Instruct-2507 | ~3 GB | **是** | 走云端 |
| **高端** | Qwen3.6-35B-A3B | ~21 GB | 否(用户主动启用) | 本地够用 |

**默认下载策略**:工具首次启动只下载 Qwen3-4B,~2.5 GB,5-10 分钟搞定。其他档位作为高级选项,用户主动启用时再下载。

## 8. 任务-档位匹配矩阵

| 任务 | 极轻量 | 默认 4B | 高端 35B-A3B |
|---|---|---|---|
| 字幕处理(改错字、调标点) | ✓ | ✓ | ✓(过剩) |
| 中英互译(常规字幕) | △ | ✓ | ✓ |
| 西/法/德 ↔ 英 | ✗ | ✓ | ✓ |
| 俄语翻译 | ✗ | △ | ✓ |
| 长视频分段(<30 分钟) | ✗ | ✓ | ✓ |
| 长视频分段(>1 小时) | ✗ | △ | ✓ |
| 热点段落分析 | ✗ | △ | ✓ |
| 专业术语翻译 | ✗ | △ | △ → 走云端 |

✓ = 适合;△ = 能跑但质量不稳定;✗ = 不要尝试

**对工具的设计含义**:
- 默认 4B 可以勾上前 5 个任务作为"开箱可用"
- 后 3 个任务在 UI 上明确标注"建议高端档或云端"
- 用户感知上,本地工具是"免费的基础能力",高质量是"按需付费/按需投入硬件"

## 9. Qwen 系列在你这个场景的优势

由于工具同时面对中文 + 多语言用户,Qwen 系列有几个隐性优势:

1. **训练数据中文占比高**——中文场景质量明显领先
2. **201 语言覆盖**(3.5 起)——比 Gemma 4 的 140+、Mistral 的 8-10 种更广
3. **Apache 2.0 一致性**——开源 size 法律链路最干净
4. **GGUF 生态完善**——Unsloth Dynamic 2.0 量化精度损失极小
5. **MoE 架构跨硬件友好**——35B-A3B 在 CPU + 大内存机器也能跑

潜在风险:
- 阿里政策不确定性,Plus / Omni 系列已闭源化,**未来 Qwen 4 / 4.5 是否继续 Apache 不能保证**
- 缓解方案:**模型加载层做成可插拔接口**,以后切 Gemma 5 / Mistral 7 不需要重写工具

## 10. 验证清单

部署到工具默认配置前必跑:

1. **字幕处理基准**(最容易验证)
   - 准备 100 条含错别字、错标点的 ASR 原文
   - 跑 4B 处理,人工抽查 20 条
   - 通过率 > 90% 即合格
2. **简单翻译质量**
   - 准备 50 条字幕(中英各 25)
   - 4B 翻译,人工打 1-5 分,平均 ≥ 3.5 即合格
3. **不同硬件下的速度实测**
   - RTX 3060 / 4060 / Apple M2 / CPU 这四档各跑一次 1 小时字幕翻译
   - 记录中位数耗时,作为工具 UI 上的"预计耗时"参考
4. **第一次加载延迟**
   - 冷启动 → 首个 token 输出的时间
   - 给用户进度提示,避免误以为卡死

## 11. 不推荐的方向(避坑)

- **Llama 4**:EU 禁用,不要碰
- **Qwen3-9B / Qwen3.5-9B**:速度不及 35B-A3B,质量不及 27B Dense,**适用区间空洞**
- **Gemma 1/2/3**:协议复杂,新工具直接上 Gemma 4
- **基于 Qwen 微调的小众变体**:协议链路不清晰,谨慎
- **专门翻译模型**(NLLB-200、SeamlessM4T):覆盖语言更广,但通用 LLM 在你这个场景已经够,不必额外维护

---

## 12. 最终建议

**默认推荐**:**Qwen3-4B-Instruct-2507**(Apache 2.0,Q4_K_M 量化 ~3 GB,128K 上下文)。

理由:
- 协议干净
- 显存需求覆盖 70%+ 用户硬件
- 字幕处理 / 简单翻译 / 短 transcript 处理 完全够用
- 下载快、加载快、推理快、用户体验顺
- 复杂任务的能力缺口由云端 API(另文档)补齐

**让用户能切换的档位**:
- Qwen3-1.7B / Gemma 4 E2B(老硬件兜底)
- Qwen3.6-35B-A3B(24GB+ 用户进阶)

**绝对不要给用户的选项**:
- Llama 4(EU 法律风险)
- Gemma 3 及以下(自定义协议)
- 9B 系列(无清晰最优区间)

**架构设计原则**:
- 模型加载层做成可插拔的 OpenAI 兼容接口
- 默认本地模型 + 可选切换高端本地模型 + 可选启用云端 API
- 每个任务在 UI 上标注"推荐档位",对超出本地能力的任务明确引导用户走云端
