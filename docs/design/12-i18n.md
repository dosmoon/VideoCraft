# 12 — 本地化（i18n）

## 动机

VideoCraft 的第一波开源用户预计来自英文社区（Reddit、GitHub、Discord），但核心开发和早期用户是中文创作者。一套 UI 同时服务两类用户，通过 `tr(key)` 查表 + 用户可切换的语言偏好实现，无需维护两份代码。

## 核心文件

| 文件 | 职责 |
|------|------|
| [src/i18n.py](../../src/i18n.py) | `tr(key, **kwargs)` + `get_current_lang()` + `set_current_lang(code)` + locale 加载 |
| [src/i18n/zh.json](../../src/i18n/zh.json) | 中文 locale 表 |
| [src/i18n/en.json](../../src/i18n/en.json) | 英文 locale 表（factory default） |
| [src/tools/preferences/preferences.py](../../src/tools/preferences/preferences.py) | `PreferencesApp` Tab 工具，用户在这里切换语言 |
| `<repo>/user_data/settings.json` | 持久化用户偏好（`{"language": "en" \| "zh", ...}`）。绿色便携后落 user_data/，老 `~/.videocraft/` 首次启动自动迁移 |

## tr() 语义

```python
tr('menu.file')                              # -> "File" 或 "文件"
tr('log.download.success', filename='a.mp4') # -> "Download complete: a.mp4"
tr('not.yet.defined.key')                    # -> "not.yet.defined.key"（兜底）
```

**Fallback 链**（三级）：
1. 当前语言表里有这个 key → 返回翻译
2. 当前语言表里没这个 key → 查 factory default 表（`en`）
3. factory default 表里也没这个 key → 返回 key 字符串本身（防崩，同时是"未定义 key"的可见信号）

`tr()` 支持 `str.format(**kwargs)` 参数插值，适合带动态内容的字符串（文件名、进度百分比等）。插值失败（key 不匹配）时静默降级为不插值文本，不崩。

## Key 命名约定

**flat dot-notation**，不用嵌套 JSON：

```
menu.file                      主菜单 label
menu.file.open_folder          菜单项
hub.welcome.title              Hub 欢迎页元素
hub.log.title                  底部日志面板
dialog.xxx.yyy                 通用对话框（目前几乎没用到）
tool.<tool_key>.xxx            具体工具内的字符串
status.xxx                     状态栏/进度显示
error.xxx                      错误信息模板（带 {e} 占位符）
log.xxx                        日志条目模板
```

**规则**：
- **所有 key 都是纯 ASCII**——即使是中文 UI 的 key，也写成 `menu.file` 而不是 `菜单.文件`，便于代码里搜索替换
- **工具 key 用 `tool.<tool_key>.xxx`**，`<tool_key>` 就是 [TOOL_MAP](../../src/VideoCraftHub.py) 里的注册 key（例如 `tool.translate.btn_start`、`tool.preferences.save`）
- **不重复**：同一个英文词（例如"Save"）在不同场景应该用不同 key（`tool.preferences.save` vs `dialog.common.save`），避免将来翻译歧义

## 持久化 + 切换流程

```
用户点 File > Preferences > 保存
    ↓
i18n.set_current_lang("en")
    ↓
写 <repo>/user_data/settings.json: {"language": "en"}
    ↓
PreferencesApp 显示橙色提示"已保存，关闭重启以生效"
    ↓
用户关闭 Hub → 重新启动
    ↓
i18n 模块 import 时 _CURRENT = get_current_lang() → 读到 "en"
    ↓
_TABLE = _load_locale("en")
    ↓
所有 tr() 调用返回英文
```

**为什么重启而非热切换**：Tk 的 Label 文本在 `tk.Label(text=...)` 创建时就固化，想改必须 `.config(text=new)` 或 `.destroy()` 重建。要热切换就得：
- 维护"每个 widget 对应哪个 key"的注册表
- 切换时遍历注册表重新 `.config()`，或者整个窗口销毁重建
- 菜单（`tk.Menu`）的 label 更麻烦，需要 `.entryconfigure(index, label=...)`

工程量远大于"关一下再开"。当前 Hub 启动只要 1 秒，代价可接受。

## Factory Default：English

`DEFAULT_LANG = "en"`。理由：
- 开源分发到 GitHub/Reddit 后，首波新用户大概率是英文用户
- 已有中文用户一次切换到 zh 后会记住，不影响
- 英文 fallback 也意味着新增字符串时：开发者可以先加英文 key，中文翻译来不及也不会崩

fallback 链里 factory default 不只是"初始值"，也是"权威源表"——缺 key 时 fallback 到它。

## 不做

- **gettext / .po 格式**——过度工程，50~500 条 flat JSON key 足够
- **动态切换**——Tk 限制，重启成本低
- **第三语言**（日/韩/西等）——先把中英两种做好，有实际需求再加
- **机器翻译自动化**——en.json 先手工翻译，Phase 8 再考虑用 AI 批量补齐
- **按模块 lazy load locale 表**——两张表各 ~100 条，全量 import 时读完没有性能问题
